# 优化后端选型指南

> 综合对比 SLAM 领域三大主流非线性优化后端：Ceres Solver、g2o 和 GTSAM，从 API 风格、求解器能力、流形支持、边缘化策略到生态适配的全景分析。

## 一、三大后端对比总表

| 维度 | [[组件-Ceres-Solver\|Ceres Solver]] | g2o | [[组件-GTSAM\|GTSAM]] |
|------|--------------------------------------|-----|------------------------|
| **开发方** | Google | U. Freiburg | Georgia Tech (Borg Lab) |
| **版本** | 2.2+ (主动维护) | 2020 分支 (低活跃) | 4.3a1 (主动维护) |
| **语言** | C++ (为核心), Python bindings (实验性) | C++ | C++ (为核心), Python (成熟 wrappers), MATLAB |
| **API 风格** | CostFunction + Problem 显式建模, AutoDiff/SelfAdjointDiff/NumericDiff | BaseEdge/BaseVertex 继承, 显式图结构 | Factor Graph 语义: NonlinearFactorGraph + Values, 概率模型内建 |
| **核心抽象** | 残差块 (ResidualBlock) + 参数块 (ParameterBlock) | 边 (Edge) + 顶点 (Vertex) + 图 (SparseOptimizer) | 因子 (Factor) + 变量 (Values) + 图 (NonlinearFactorGraph) |
| **求解器** | LM, Dogleg, TrustRegion; DENSE_QR, SPARSE_NORMAL_CHOLESKY, ITERATIVE_SCHUR, CGNR | GN, LM, Dogleg; CHOLMOD, CSPARSE, PCG | LM, GN, Dogleg; 通过 Eigen/SuiteSparse 和 ordering 策略控制 |
| **流形支持** | Manifold (4.0+ 替代 LocalParameterization), QuaternionManifold, SubsetManifold | 需自定义 `oplusImpl` (手写 tangent update) | 内建 Lie group: Pose2, Pose3, Rot3, NavState 等, `retract`/`localCoordinates` |
| **边缘化** | 需手写实现 (Schur complement, marginalization prior) | 需手写实现 | ISAM2 Bayes 树增量可逆边缘化; 固定滞后平滑 FixedLagSmoother |
| **增量求解** | 无原生支持 (需用户管理滑窗 + marginalization prior) | 无原生支持 | iSAM2: Bayes 树增量 + 选择性重线性化 |
| **信息/协方差** | 需从正规方程或 Hessian 手动恢复 | 需手算 Hessian 逆或 Cholesky 抽取 | Marginals: 自动边缘协方差/信息矩阵, API: `marginalCovariance(key)` |
| **鲁棒核** | Huber, Cauchy, Tukey, SoftLOne, Arctan, TrivialLoss | Huber, Cauchy, DCS, Tukey | Huber, Cauchy, GM; 通过 `noiseModel::Robust` 封装 |
| **自动求导** | AutoDiff (C++) + Jet dual numbers, 支持 Eigen; Python: CostFunction wrapper | 无 AutoDiff, 需手写 Jacobian 或数值微分 | Python `CustomFactor` 需手写 Jacobian; C++ 可通过 `ExpressionFactor` 半自动求导 |
| **生态 / 在 SLAM 中的应用** | VINS-Fusion, [[算法-CT-ICP]], [[算法-Cartographer]], DSO, SVO Pro | [[算法-ORB-SLAM3]], [[组件-OpenMAVIS]], RTAB-Map | [[算法-LIO-SAM]], [[算法-Kimera-VIO]], [[算法-DM-VIO]], [[算法-FAST-LIO-SAM-SC-QN]], [[组件-gtsam_points]] |
| **依赖体量** | 轻量 (Eigen + SuiteSparse/glog/gflags) | 中等 (Eigen + SuiteSparse + CSPARSE) | 较重 (Boost + Eigen + TBB + SuiteSparse/Metis + Python wrapper) |
| **ROS 集成** | 无 ROS 依赖, ceres-solver 为独立库 | ROS 生态老去, g2o 常作为 ROS dependency 存在 | 无 ROS 依赖, 但 ROS 中 GTSAM 通过 catkin/colcon 集成 |
| **Python 支持** | 实验性, 不稳定 | 无原生 Python | 成熟 pybind11 wrappers, `pip install gtsam` 可用 |
| **文档质量** | 出色: 教程 + 官方文档 + 代码注释 | 适中, 多依赖社区 | 优秀: 完整 Python 文档 + C++ Doxygen + 大量官方 examples |

## 二、核心能力深度对比

### 2.1 API 风格与建模心智模型

| 心智模型 | Ceres | g2o | GTSAM |
|----------|-------|-----|-------|
| **问题定义** | 添加 CostFunction 到 Problem | 添加 Vertex + Edge 到 SparseOptimizer | 添加 Factor 到 NonlinearFactorGraph, 初始化 Values |
| **变量管理** | 显式 parameter block 指针 (raw pointer) | Vertex 继承, id 管理 | Key/Symbol 索引, `Values` 容器强类型 |
| **噪声建模** | 通过 `LossFunction` (鲁棒核) 和隐式的 `sqrt_info` 加权 | 通过 `InformationEdge` / `RobustKernel` | `noiseModel` 对象: Diagonal, Isotropic, Unit, Constrained, Robust, Gaussian |
| **gauge freedom** | 需手动固定变量 (SetParameterBlockConstant) | 需手动固定 Vertex (setFixed) | 通过 PriorFactor 或 NonlinearEquality 软/硬约束 |
| **代码量** | 中等: CostFunction 定义 + Problem 填充 + Solve | 多: Vertex/Edge 子类 + optimizer 配管 | 少: 因子图建模最简洁, 增量更新最省代码 |

**建模示例：Pose3 PGO 添加一条 odometry 边**

```python
# GTSAM — 最简洁
import gtsam
from gtsam.symbol_shorthand import X
graph = gtsam.NonlinearFactorGraph()
graph.add(gtsam.BetweenFactorPose3(X(0), X(1), odom_pose, noise))
# isam.update(graph, values)  — 增量更新
```

```cpp
// Ceres — 需显式定义 CostFunction 和 parameter blocks
auto* cost = new PoseGraph3dErrorTerm(p_a, p_b, q_a, q_b, sqrt_info);
ceres::Problem problem;
problem.AddResidualBlock(cost, nullptr, pose0, pose1);
// Solve — 批量求解, 无原生增量
```

### 2.2 求解器能力

| 求解器 | Ceres | g2o | GTSAM |
|--------|-------|-----|-------|
| **Levenberg-Marquardt** | ✓ (默认, lambda 调参丰富) | ✓ | ✓ |
| **Gauss-Newton** | ✓ | ✓ | ✓ |
| **Dogleg** | ✓ (TrustRegion) | ✓ | ✓ |
| **Graduated Non-Convexity** | — | — | ✓ (`GncOptimizer` for robust non-convex costs) |
| **线性求解器** | DENSE_QR, SPARSE_NORMAL_CHOLESKY, SPARSE_SCHUR, ITERATIVE_SCHUR, CGNR | LinearSolverPCG, LinearSolverCSparse, LinearSolverCholmod | 通过 Eigen/SuiteSparse + `Ordering` 策略选择, 内部不暴露求解器类型 |
| **多线程** | OpenMP 并行 `AddResidualBlock` / Jacobian 填充 | 有限 | TBB 并行和 ISAM2 增量更新内部并行 |
| **Incremental (在线)** | 无原生 | 无原生 | ISAM2 原生增量, Bayes tree 自动变量排序 |

### 2.3 流形 (Manifold) 支持

SLAM 中位姿、旋转、IMU 状态等都定义在 Lie group / manifold 上，线性化时必须使用正确的切空间更新。三者的流形支持差异是选型的关键因素：

| 能力 | Ceres | g2o | GTSAM |
|------|-------|-----|-------|
| **SO(3) / SE(3)** | Manifold (Ceres 4.0+), `QuaternionManifold` | 需手写 `oplusImpl` | `Rot3`, `Pose3` 内建 Lie group: `retract(delta)`, `localCoordinates(q)` |
| **NavState (姿态+位置+速度)** | 需手动拆分参数块 | 需手写 Vertex | `NavState` 内建: tangent `[dR, dP, dV]`, `Expmap/Logmap` |
| **IMU bias** | VectorManifold 或直接欧式 | 欧式 Vertex | `imuBias::ConstantBias` + bias factor family |
| **逆深度 / bearing vector** | SubsetManifold | 需手写 `oplusImpl` | 通过 Factor 隐式处理或手写 |
| **自定义 manifold** | `AutoDiffManifold` / `Manifold` 子类 | `BaseVertex::oplusImpl` | 继承 `LieGroup` / `Manifold` traits |
| **右扰动 / 左扰动约定** | 用户自行控制 | 用户自行控制 | 默认 **右扰动** (GTSAM `ExpMap` = right exponential) |
| **Jacobian order** | 列优先 (Eigen default) | 用户自行管理 | **Fortran order (column-major)**, Python callback 写 `H[i]` 时注意 |

> GTSAM 的右扰动约定 (`x_new = x * Exp(delta)`) 需要特别注意: 如果从左扰动推导 (`x_new = Exp(delta) * x`)，Jacobian 的映射关系会不同。Ceres Manifold 允许用户选择 Plus/Jacobian 实现，更灵活但易出错。

### 2.4 边缘化能力

边缘化 (marginalization) 是 SLAM 后端最棘手的问题之一。三者能力差距极大：

| 能力 | Ceres | g2o | GTSAM |
|------|-------|-----|-------|
| **手动 Schur complement** | 需从 Problem 提取 Hessian 后手写消元 | 需手写实现 | ISAM2 Bayes tree 自动处理 |
| **增量可逆边缘化** | 不可逆 | 不可逆 | ISAM2 可在 Bayes tree 中逆操作, 通过添加反因子重线性化 |
| **Fixed-lag smoothing** | 需自行管理窗口 + prior factor | 需自行管理 | `FixedLagSmoother` 内建 (当前未在 Python 暴露) |
| **Smart factor (隐式边缘化路标)** | 需手写内部 Schur complement | 需手写 | `SmartProjectionFactor` / `SmartStereoFactor` 内建 |
| **边缘化先验维护** | 需手写稠密 prior block + 线性化点固定 | 需手写 | ISAM2 内部管理, DM-VIO 的延迟边缘化是此能力的典型工程应用 |
| **选择性重线性化** | 需重存原始观测再优化 | 需重存原始观测 | ISAM2 对受影响变量自动选择性重线性化 (Bayes 树 |

## 三、在 SLAM 系统中的选用分布

| SLAM 系统 | 后端 | 选择原因 |
|-----------|------|----------|
| VINS-Fusion | **Ceres** (DENSE_SCHUR + DogLeg) | 滑窗优化, 需要 DENSE_SCHUR 对窗口内稠密 Hessian 高效; CostFunction 灵活定义 IMU+视觉+先验残差; 自动求导加速开发 |
| [[算法-Cartographer]] | **Ceres** (scan-to-submap + 后端 PGO) | 本地匹配用 Ceres 构建残差; 概率网格匹配的自动求导 CostFunction 灵活; 后端需要 SPARSE_NORMAL_CHOLESKY |
| [[算法-CT-ICP]] | **Ceres** (OpenMP 多线程并行) | Ceres 的 OpenMP 并行 Jacobian 对大量 KNN 邻域计算有利; 四种残差类型 (POINT_TO_PLANE / POINT / LINE / DISTRIBUTION) 统一在 Problem 中表达 |
| [[算法-ORB-SLAM3]] | **g2o** (多级 BA + Sim3) | 历史原因, ORB-SLAM 系传统选择 g2o; 多级 BA (Motion / Local / Essential Graph / Full) 通过 g2o 图结构管理; Atlas 多地图需要 g2o 的 SparseOptimizer 分离 |

| [[算法-Kimera-VIO]] | **GTSAM iSAM2** (FixedLagSmoother) | SmartStereoFactor 隐式消路标; Fixed-lag smoother 管理窗口; 视觉因子图表达清晰 |
| [[算法-DM-VIO]] | **GTSAM iSAM2** (延迟边缘化) | ISAM2 Bayes 树支持延迟边缘化 (等待参数收敛); 桥接 DSO 视觉 Hessian 和 GTSAM IMU 因子 |
| [[算法-FAST-LIO-SAM-SC-QN]] | **GTSAM ISAM2** (PGO) | 后端 PGO, 回环因子 + BetweenFactor + PriorFactor, ISAM2 增量更新 |
| [[phad_fusion设计总结\|phad_fusion]] | **GTSAM iSAM2** | "Everything is a Factor" 哲学: LiDAR ICP + 视觉重投影 + IMU 预积分 + GNSS + 回环 → 统一因子图 |

## 四、选型决策指南

### 4.1 选 Ceres 的场景

适合 Ceres 的场景：
- **滑动窗口优化**：滑窗内变量数目有限 (通常 10-40 帧)，Schur complement 手写后可精确控制边缘化先验格式。
- **需要自动求导快速原型**：Ceres 的 AutoDiff Jet 是为 C++ 最成熟的自动求导方案，对自定义 CostFunction 的 Jacobian 推导免去了手写和数值验证开销。
- **局部配准 / 扫描匹配**：Cartographer 的 scan-to-submap、CT-ICP 的配准、SVO Pro 的 BA 都在局部问题上使用 Ceres。
- **轻量依赖**：Ceres 依赖链最轻 (Eigen + SuiteSparse + glog)，适合对编译时间和包体积敏感的项目。
- **自定义求解器策略**：DENSE_SCHUR / ITERATIVE_SCHUR 等线性求解器的选择空间大，适合对速度有极致要求的场景。

不适合 Ceres 的场景：
- **长期在线增量 SLAM**：无原生 ISAM2 等价物，需自行管理图结构、变量排序、增量更新和边缘化。
- **多传感器因子图融合**：添加新传感器需要新增 CostFunction 并手动管理窗口/边缘化，无因子图语义。
- **需要边际协方差查询**：需从 Hessian 手动恢复，不支持 `marginalCovariance(key)`。

### 4.2 选 GTSAM 的场景

适合 GTSAM 的场景：
- **增量在线 SLAM**：iSAM2 是唯一同时满足"实时+因子图建模+接近批量精度+可逆边缘化+自动协方差恢复"的方案。
- **多传感器因子图融合**：LiDAR、视觉、IMU、GNSS、轮速计等统一为因子，传感器可插拔，无需重新推导耦合雅可比。
- **需要自动协方差/不确定性查询**：`Marginals` 和 ISAM2 的 `marginalCovariance(key)` 自动恢复边缘不确定性。
- **"Everything is a Factor" 架构**：如 phad_fusion，每次观测是独立因子，噪声模型与因子一一对应，回环即 BetweenFactor。
- **Python 原型开发**：GTSAM 的 Python wrapper 是三家中最成熟稳定的，`pip install gtsam` 可直接使用。
- **视觉因子内建**：SmartProjectionFactor、GenericStereoFactor、GenericProjectionFactor 等视觉因子开箱即用。

不适合 GTSAM 的场景：
- **需要极细粒度的求解器控制**：GTSAM 不暴露线性求解器类型选择，对大图 ordering 策略和 solver 细节的控制不如 Ceres 直接。
- **极轻量依赖**：GTSAM 需要 Boost + TBB + Metis 等，对嵌入端和最小化依赖场景不友好。
- **C++ 自动求导需求**：GTSAM 的 `ExpressionFactor` 半自动求导不如 Ceres AutoDiff 成熟；Python `CustomFactor` 手写 Jacobian 并需要 Fortran order。

### 4.3 选 g2o 的场景

适合 g2o 的场景：
- **ORB-SLAM 系列兼容**：如果项目基于 ORB-SLAM3 或需要 ORB-SLAM 的 Atlas/BA 体系，g2o 是唯一选择。
- **需要明确图结构暴露**：g2o 的 Vertex/Edge 继承体系对教学和理解图优化内部机制非常友好。
- **历史项目维护**：早期 SLAM 项目大量使用 g2o，维护旧代码时不宜更换后端。

不适合 g2o 的场景：
- **新项目启动**：g2o 社区活跃度低，API 设计老旧 (C++03 风格, 大量裸指针)，无 Python 支持，无增量求解。
- **现代工程需求**：无自动求导, 无原生流形 (需手写 `oplusImpl`), 无 native 边缘化, 缺少自动协方差查询。
- **多传感器融合**：没有因子图语义, 每个新传感器需从 Vertex/Edge 继承重新设计。

## 五、混合使用策略

在实际工程中, 最佳设计往往不是"只选一个", 而是分层搭配：

| 层 | 推荐后端 | 理由 |
|----|----------|------|
| **局部配准 / 扫描匹配** | Ceres | AutoDiff 快速原型, 求解器选择多 |
| **滑动窗口局部 BA** | Ceres 或 GTSAM FixedLagSmoother | 需要 DENSE_SCHUR 时用 Ceres, 需要 Smart Factor 时用 GTSAM |
| **在线增量全局后端** | GTSAM iSAM2 | 增量 + 协方差 + 因子图语义 |
| **回环后全局一致性** | GTSAM PGO 或 g2o | GTSAM 的 BetweenFactor + PGO 简洁; ORB-SLAM 体系用 g2o Essential Graph |
| **多传感器因子融合** | GTSAM | 统一因子图, 传感器可插拔 |

典型混合架构参考 (phad_fusion)：

```
传感器层 → 预处理/前端 (C++ 核, 无后端依赖)
          ├→ LiDAR ICP 因子 (自定义, 调用 Ceres 做局部匹配)
          ├→ 视觉投影因子 (GTSAM SmartProjectionFactor)
          ├→ IMU 预积分因子 (GTSAM CombinedImuFactor)
          ├→ GNSS 先验因子 (GTSAM PriorFactor<Pose3>)
          └→ 回环因子 (GTSAM BetweenFactor<Pose3>)
                    ↓
              GTSAM ISAM2 统一增量求解
                    ↓
              输出: Values → 位姿/速度/偏置; Marginals → 协方差
```

## 六、性能与调参

### 6.1 计算性能定性比较

| 场景 | Ceres | g2o | GTSAM iSAM2 |
|------|-------|-----|-------------|
| 批优化 (10-100 变量) | 最快 (AutoDiff 开销小) | 接近 | 因子图 overhead 稍高 |
| 批优化 (1000+ 变量) | SPARSE_SCHUR 高效 | 渐慢 | ISAM2 ordering 效果相近 |
| 增量添加 1 帧 (在线) | 需完整批优化 | 需完整批优化 | ISAM2.update() 毫秒级 |
| 协方差查询 (单个变量) | 需 Hessian 全求逆 | 需 Hessian 全求逆 | Marginals 局部 Cholesky O(1) |
| 回环后全局重优化 | 批优化 | 批优化 | ISAM2 增量 (只影响受回环影响的子树) |

### 6.2 关键调参

**Ceres**
- `linear_solver_ordering`: 大图时手动设定参数块消元顺序, 影响 Schur complement 填充 (fill-in)
- `max_num_iterations`: 通常 10-20; 批优化收敛快
- `function_tolerance`: 过小会浪费迭代
- `trust_region_strategy`: LM 默认; 大残差初值时 Dogleg 可能更快

**GTSAM ISAM2**
- `ISAM2Params::relinearizeThreshold`: 控制重线性化激进程度, 默认 0.1 (小 → 频繁重线性化)
- `ISAM2Params::relinearizeSkip`: 每次 update 后跳过多少次再检查重线性化, 默认 1 (每次检查)
- `ISAM2Params::enableRelinearization`: 必须开启
- `ISAM2Params::cacheLinearizedFactors`: 开启则缓存线性化因子, 重线性化时不需要重新计算
- 定期调用 `ISAM2::update()` 时传入的 Values 的初值质量直接影响收敛

**g2o**
- `SparseOptimizer::setVerbose(true)` 查看每步 error 变化
- `optimizer.initializeOptimization()` 前后检查图结构
- 单次 `optimize(1)` 迭代配合自定义终止条件
- Schur complement 是否开启 (`setComputeBatchStatistics`)

### 6.3 环境与编译

| | Ceres | g2o | GTSAM |
|---|-------|-----|-------|
| Ubuntu 安装 | `apt install libceres-dev` | `apt install libg2o-dev` | `apt install libgtsam-dev` + pip |
| CMake 集成 | `find_package(Ceres REQUIRED)` | `find_package(g2o REQUIRED)` | `find_package(GTSAM REQUIRED)` |

| AArch64 / 嵌入 | 良好支持 | 一般 | 较重, 但官方 CI 已覆盖 |

## 七、实战建议：调参与排错

### 7.1 通用调试顺序

无论使用哪个后端，非线性优化排障的检查清单都类似：

1. **因子/残差计数**：确认 `graph.size()` 或 `problem.NumResidualBlocks()` 是否等于预期。漏因子是最高频错误。
2. **单因子 error 检查**：对每个因子调用 `unwhitenedError(values)` 或 `cost_function->Evaluate(parameters, residuals)` 看残差模长是否合理 (通常 < 1.0, 否则参数单位/坐标系有问题)。
3. **噪声模型校验**：残差维度 = noise model 维度，协方差/信息矩阵正定，sigma 数量级合理 (不要用 1e-12 掩盖建模错误)。
4. **优化前后 error 对比**：`error(initial)` / `error(result)` 应下降 2-6 个数量级。如果只下降 10-20%, 存在 gauge freedom 未锚定或初值太差。
5. **初值合理性**：位姿初值应在真值 0.5 m/30° 以内, 否则非线性优化可能收敛到错误局部极小。
6. **自定义因子数值 Jacobian 验证**：任何手写 Jacobian 都必须用 `numericalDerivative` 验证 (rtol 1e-5, atol 1e-8)。

### 7.2 各后端的独有坑

| 坑 | 后端 | 现象 | 对策 |
|----|------|------|------|
| 参数块指针混叠 | Ceres | Segfault 或 Jacobian 全零 | 确认每个 parameter block 的地址唯一 |
| AutoDiff with non-trivial manifold | Ceres | 梯度方向错误 | 4.0+ 用 Manifold 而非 LocalParameterization, 确认 Plus/Jacobian 正确 |
| Column-major order | GTSAM | Python callback Jacobian 输出方向错误 | `np.zeros((r, c), order='F')` |
| Right-perturbation convention | GTSAM | 自导 Jacobian 和 GTSAM 内建因子符号相反 | 自导 Jacobian 使用 `d r(x*Exp(eps)) / d eps` |
| Gravity direction mismatch | GTSAM | IMU 预积分残差巨大, 尺度错误 | 统一 ENU (`MakeSharedU`) 或 NED (`MakeSharedD`), 全图一致 |
| 边缘化先验 Hessian 不对称 | Ceres/g2o | 滑窗优化后先验残差上升 | Schur complement 实现中 `H_marg = H_new - H_cross^T * H_inner^{-1} * H_cross`, 检查矩阵维度 |
| g2o Vertex ID 冲突 | g2o | 图重建崩溃 | `SparseOptimizer::save` / `load` 时确认 vertex ID 持续递增 |
| ISAM2 不更新 | GTSAM | `isam.update()` 后估计不变 | 检查是否有新因子 (重复 update 空图不会触发重优化), 检查 `relinearizeThreshold` 是否过小 |

### 7.3 性能优化的工程策略

- **因子批量化**：Python GTSAM 中多个同类 measurement 合并为一个 CustomFactor 处理多条测量 (`H[i]` 多列), 减少 Python↔C++ callback 次数。
- **Ceres AutoDiff 的成本**：AutoDiff Jet 引入额外算力开销 (dual number 比 float 慢 ~2-5×)。对性能关键路径的 CostFunction 用手写 analytic Jacobian。
- **ISAM2 的 `update()` 频率**：不需要每个 sensor tick 就 `update()`。IMU 预积分降频到 10-30 Hz, GNSS 1 Hz, 回环按需 —— 只在有新因子时才调用 `update()`。
- **重线性化策略**：ISAM2 默认 `relinearizeSkip=1` (每次检查), 对图大部分稳定的场景可增大至 5-10, 显著减少重线性化开销。
- **Ceres linear solver ordering**：大图 (>500 参数块) 时 `SPARSE_SCHUR` 的 ordering 对 fill-in 和求解速度影响极大, 建议使用 `AMD` 或 `NESDIS` ordering。

### 7.4 Python vs C++ 性能差异

| 操作 | GTSAM Python | GTSAM C++ | 差距 |
|------|-------------|-----------|------|
| `CustomFactor` (10K 条) 单次 linearize | ~500 ms | ~20 ms | ~25× |
| `ISAM2.update()` (500 var) | ~10 ms | ~3 ms | ~3× |
| `Marginals` 10 变量 | ~5 ms | ~1 ms | ~5× |
| 建图/配置开销 | 最小 | 最小 | 相当 |

> Python CustomFactor 的 callback 每次 linearize 都跨 Python↔C++ 边界 (pybind11 + GIL), 适合原型验证和少量 measurement。在线部署的关键路径应搬 C++ factor。

## 八、总结与推荐

| 项目类型 | 推荐后端 | 一句话理由 |
|----------|----------|-----------|
| 新项目——在线 SLAM / 多传感器融合 | **GTSAM** | 因子图语义 + iSAM2 增量 + 自动协方差 + Python 原型, 全栈能力最强 |
| 新项目——滑窗优化 / 局部配准 | **Ceres** | AutoDiff 最成熟, 求解器选择多, 依赖轻, 工程灵活性最高 |
| 新项目——教学 / 理解图优化 | **g2o 或 GTSAM** | g2o 暴露图内部; GTSAM Python + 文档更适合快速学习 |
| 维护 ORB-SLAM / 旧项目 | **g2o** | 已有生态, 更换代价大于收益 |
| 内嵌 / MCU 或极小依赖 | **Ceres** | 依赖最轻, 编译快 |
| Python 优先 / 研究原型 | **GTSAM** | Python wrapper 成熟, `pip install gtsam` 即用 |
| 批量大规模 BA / SfM | **Ceres** | SPARSE_SCHUR + AutoDiff 对大批量问题效率最高 |

> **核心建议**：如果没有历史包袱, 新 SLAM 项目的最佳组合是 **Ceres 做局部配准和前端优化 + GTSAM iSAM2 做全局后端因子图融合**。这种分层架构既利用 Ceres 的 AutoDiff 灵活性和求解器多样性, 又获得 GTSAM iSAM2 的增量更新与自动协方差恢复。
