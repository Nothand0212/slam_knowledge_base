# 关键帧选择策略

> 关键帧选择决定了后端优化图的信息密度和计算负载。三个代表性 SLAM/VIO 项目分别使用视差驱动、多条件复合、和 iSAM2 同架构策略，覆盖了工程中主要的 KF 设计范式。

## 问题背景

滑动窗口/固定滞后优化器中，KF 率直接决定：
- **信息密度**：太稀疏 → 约束不足，精度下降
- **计算负载**：太密集 → 优化规模膨胀，实时性崩溃
- **线性化点**：iSAM2/滑窗中的线性化点一旦固定就难以更新，密度过高的 KF 会把线性化误差永久锁入系统

## 三种策略对比

| 维度 | VINS-Fusion | ORB-SLAM3 | Kimera-VIO |
|------|------------|-----------|------------|
| 后端 | Ceres 滑窗（10帧） | g2o 局部/全局 BA | GTSAM iSAM2 固定滞后（25-30状态） |
| 决策函数 | `addFeatureCheckParallax()` | `NeedNewKeyFrame()` | `shouldBeKeyframe()` |
| 核心指标 | 归一化视差 | inlier 比例 vs 参考KF | 视差 + 时间 + 特征数 |
| KF 率 | ~15-20% | ~10-25%（按传感器调整） | ~25% |
| 冗余剔除 | 无（边缘化自动完成） | KeyFrameCulling（90% 覆盖剔除） | 无（固定滞后自动完成） |
| 参数自适应 | 无（固定阈值） | thRefRatio 按传感器/IMU 动态 | 无（固定阈值） |

## VINS-Fusion：视差驱动 + 质量守卫

**源码锚点**: `raw/codes/VINS-Fusion/vins_estimator/src/estimator/feature_manager.cpp:L52-L119`

### 决策逻辑

```
addFeatureCheckParallax(frame_count, image):
  1. 质量守卫（立即触发 KF）:
     if frame_count < 2           → true   # 窗口初期
     if last_track_num < 20       → true   # 跟踪特征不足
     if long_track_num < 40       → true   # 长期跟踪特征少
     if new_feature_num > 0.5 * last_track_num → true  # 场景突变

  2. 视差检查（跨倒数第二帧与倒数第三帧）:
     for each feature tracked across these two frames:
       parallax = compensatedParallax2(feature)
     avg_parallax = sum / count
     if parallax_num == 0        → true    # 无跨帧特征
     return avg_parallax >= MIN_PARALLAX   # 默认 ~10/460 = 0.022 (归一化)
```

### 关键设计决策

1. **视差在倒数第二/三帧之间计算**，不是当前帧，因为当前帧位姿尚未优化
2. **KF 决定边缘化策略**：
   - KF → `MARGIN_OLD`：删除最老帧，将其信息边缘化到先验
   - 非 KF → `MARGIN_SECOND_NEW`：删除次新帧（保留最老帧维持视差基线），丢弃次新帧观测
3. **`compensatedParallax2`**：用 IMU 旋转补偿纯旋转视差，避免旋转被误判为平移视差

## ORB-SLAM3：多条件 OR + 冗余剔除

**源码锚点**: `raw/codes/ORB_SLAM3/src/Tracking.cc:L3064-L3214` (选择) + `src/LocalMapping.cc` (剔除)

### 决策逻辑（最终表达式）

```
need_KF = ((c1a || c1b || c1c) && c2) || c3 || c4
```

| 条件 | 含义 | 典型阈值 |
|------|------|---------|
| c1a | 距上一 KF 帧数 ≥ mMaxFrames | **fps 值** (10–30) |
| c1b | 帧数 ≥ mMinFrames **且** LM 空闲 | **0 帧**（立即） |
| c1c | 跟踪极弱 (inlier < 25% refMatches) 或近距离点缺失 | — |
| c2 | inlier 比例 < thRefRatio | **0.75–0.90**（按传感器动态） |
| c3 (IMU) | 距上一 KF ≥ 0.5s | 0.5s |
| c4 (IMU mono) | 15 < inlier < 75 或 RECENTLY_LOST | — |

**thRefRatio 动态调整**：
- 系统启动时（KFs < 2）：0.4（激进建图）
- 单目：0.90（保守，防基线退化）
- 双目/IMU 双目：0.75
- IMU 单目 inlier > 350：0.75，否则 0.90

### KeyFrameCulling 冗余剔除

```
for each KF in local map:
  count how many other KFs share ≥90% of this KF's map points
  if share_ratio >= 0.9 (or 0.5 for IMU stereo):
    cull this KF

IMU safety: don't cull last 2 KFs, check time gap < 3s
```

### 设计模式总结

1. **参考关键帧概念**：inlier 比例是相对于参考 KF 的地图点数，不是绝对特征数
2. **c1a/c1b 分工**：c1a 是无条件时间上限，c1b 是 LM 空闲时的机会窗口
3. **c2 的"与"语义**：必须有定量退化证据（c2）才结合时间条件触发，避免时间到了但跟踪仍然健康的误触发
4. **传感器特定逻辑**：IMU 初始化前后、有/无 IMU、单/双/多目各自有独立的 c3/c4 逻辑

## Kimera-VIO：与 iSAM2 同架构的三条件触发

**源码锚点**: `raw/codes/Kimera-VIO/src/frontend/VisionImuFrontend.cpp:L175-L232`

### 决策逻辑

```
shouldBeKeyframe(frame, frame_lkf):
  kf_diff_ns = frame.timestamp - frame_lkf.timestamp
  disparity = computeMedianDisparity(frame_lkf, frame)

  need_KF = max_time_elapsed           // kf_diff >= 5.0s
         || max_disparity_reached      // disparity > 1000px (实质禁用)
         || disparity_flipped          // (disparity > 0.5px && min_time_elapsed >= 0.2s)
                                       // || disparity刚变低 → 运动停止时插KF
         || nr_features_low            // valid_features <= min_number_features (默认0 禁用)
         || frame.isKeyframe_          // 第一帧强制
```

### 关键设计

1. **视差翻转检测**：运动刚停止时 disparity 从高降到低于 0.5px，立即插入 KF，防止丢失关键信息
2. **min_time_elapsed 0.2s**：限制最大 KF 率 ~5Hz
3. **max_time_elapsed 5.0s**：比 VINS 和 ORB-SLAM3 都大，因为 iSAM2 的固定滞后会自动清理老状态
4. **未做主动冗余剔除**：iSAM2 的固定滞后平滑器通过边际化自动完成状态生命周期管理

## 工程模式总结

### 模式 1：视差驱动
```
if avg_parallax > threshold && time_since_last_kf > min_interval → KF
```
- 简洁，视差与信息增益直接相关
- 缺点：低纹理/纯旋转场景视差小导致 KF 不足

### 模式 2：质量守卫
```
if tracking_quality < guard_threshold → force KF
```
- 防止跟踪质量低时丢失关键帧，保护关键信息

### 模式 3：时间保护
```
if time_since_last_kf > max_interval → force KF
```
- VINS 和 Kimera 的 max_interval 相差 10 倍（0.5 vs 5.0s），反映后端架构差异

### 模式 4：冗余剔除
```
if most_points_covered_by_other_kfs → cull
```
- 只有非固定滞后架构（ORB-SLAM3）才需要主动剔除
- 滑窗/iSAM2 通过边际化自动完成

### KF 与后端的关系

| 后端类型 | 状态生命周期 | 是否需要主动剔除 | KF 率倾向 |
|---------|------------|----------------|----------|
| 固定大小滑窗 | 边缘化老状态 | 不必要 | 低~中（10-20%） |
| iSAM2 固定滞后 | 边际化老因子 | 不必要 | 中（~25%） |
| 全局 BA（ORB-SLAM3） | 手动管理 | 需要 | 中~高（10-25%） |

## Agent 实现提示

### 适用场景
需要为新的 SLAM/VIO 系统设计 KF 选择模块时，参考本页的三个工程范式和 iSAM2 特定考量。

### 输入输出契约
- **输入**：当前帧（特征、跟踪状态、时间戳）、上一关键帧引用、LM/后端状态
- **输出**：bool（是否插入关键帧）

### 实现骨架（伪代码）

```
bool shouldInsertKeyframe(current_frame, last_kf, backend_status):
    // 1. 时间保护（无条件上限）
    if time_since(last_kf) > max_interval:
        return true

    // 2. 质量守卫（防止信息丢失）
    if tracking_quality < guard_threshold:
        return true

    // 3. 视差/运动驱动（主要触发）
    if time_since(last_kf) > min_interval:
        disparity = computeDisparity(last_kf, current_frame)
        if disparity > threshold:
            return true
        // 运动停止检测
        if disparity < low_threshold && prev_disparity > low_threshold:
            return true

    // 4. 后端状态保护
    if backend_is_busy && frames_in_queue > max_queue:
        return false  // 降频，防止后端积压

    return false
```

### 实现注意事项
- VINS 的视差必须补偿 IMU 旋转分量，否则纯旋转被误判为平移
- ORB-SLAM3 的 `thRefRatio` 动态调整是传感器特定逻辑的精髓
- iSAM2 后端不需要主动冗余剔除，但需要 `max_time_elapsed` 保护
- 所有系统都区分"初始化阶段的激进 KF"和"稳定运行的保守 KF"
