# IMU预处理

> IMU 预处理指进入状态估计器之前的信号处理、标定校正、时间对齐、异常值处理和多 IMU 管理；它不包括预积分、状态传播和后端因子构建。

## 边界

IMU 数据链路可以分成两层：

- **预处理层**：把原始陀螺仪/加速度计读数变成更可信、更一致、时间戳明确的测量。
- **估计层**：用这些测量做 EKF/IESKF 传播、IMU 预积分、滑动窗口优化或因子图约束。

本页只讨论预处理层。[[概念-IMU预积分]]、[[方法-IESKF滤波器]] 和 [[IMU数据管线]] 负责估计层内容。

## 典型处理环节

| 环节 | 目标 | 常见实现 |
|------|------|----------|
| 硬件低通 | 抗混叠，抑制采样前高频噪声 | IMU 芯片寄存器配置 |
| 软件低通 | 控制进入估计器的带宽 | Butterworth、一阶/二阶 IIR |
| 陷波滤波 | 去除电机/旋翼/结构共振窄带振动 | 静态 notch、FFT 动态 notch |
| 标定校正 | 修正 scale、misalignment、offset | 六面翻滚、离线标定矩阵 |
| 温度补偿 | 修正温漂导致的 bias 变化 | 多项式温度模型、温控箱标定 |
| 时间处理 | 对齐相机/LiDAR/GNSS 时间 | 时间偏移配置、在线时延估计、插值 |
| 异常值处理 | 剔除离群 IMU 或坏数据段 | 阈值检测、多 IMU 投票 |
| 坐标统一 | 统一 body/IMU/sensor 坐标约定 | 外参、安装方向、轴向变换 |

## 工程案例对比

| 系统 | 预处理特点 | 估计器入口 |
|------|------------|------------|
| PX4 | 硬件 LPF、软件 Butterworth、静态/动态陷波、出厂标定、温度补偿、多 IMU 投票 | 姿态估计与控制环 |
| OpenVINS / MSCKF_VIO | 主要依赖离线标定和时间配置，IMU 数据进入 EKF 传播；重点在状态协方差与视觉更新 | MSCKF/EKF 传播 |
| VINS-Fusion | 使用时间偏移、外参、bias 在线估计；IMU 预处理较轻，核心在中值预积分和滑窗优化 | Ceres 滑动窗口 |
| LIO-SAM / FAST-LIO 系 | 更关注 IMU 去畸变和状态传播，通常假定输入 IMU 已完成基础标定 | LiDAR 去畸变与 LIO 传播 |
| IC-GVINS / OB_GINS | GNSS/INS 场景重视时间对齐、整秒节点、地球自转补偿和位置观测质量控制 | Ceres 滑窗组合导航 |

## PX4 作为上限参考

PX4 飞控的 IMU 预处理链路很完整：

```
IMU 芯片原始数据
  -> 硬件 LPF
  -> 软件 Butterworth 二阶 LPF
  -> 静态/动态陷波
  -> 出厂标定校正
  -> 温度补偿
  -> 多 IMU 投票
  -> 姿态估计/控制
```

这套链路直接服务飞控 PID 控制，因此对延迟、噪声和振动污染非常敏感。SLAM 系统常把 IMU 当作几何约束或传播输入，对延迟的容忍度更高，但在无人机、腿式机器人、高速车辆等高振动平台上，PX4 的滤波和陷波思路仍然值得借鉴。

## 设计建议

- 地面轮式平台：优先保证时间戳、外参、坐标系和 bias 建模正确，低通/陷波通常不是第一优先级。
- 多旋翼无人机：至少考虑电机基频附近的陷波滤波，并记录滤波引入的延迟。
- 高精度 GNSS/INS：关注 IMU 等级、温度补偿、地球自转补偿和 GNSS 时间对齐。
- 多传感器 SLAM：不要在没有建模延迟的情况下随意加重滤波；过强低通会损伤动态响应。

## 常见误区

- 把 IMU 预处理和 IMU 预积分混为一谈。预处理改变测量本身，预积分是估计器内部的约束构造。
- 只看采样率，不看有效带宽。高采样率 IMU 仍可能被机械振动污染。
- 从飞控照搬滤波参数到 SLAM。飞控控制环和 SLAM 后端的延迟预算不同，参数需要重新验证。
- 忽略时间同步。很多 IMU 问题看起来像噪声，实际是相机/LiDAR/IMU 时间错位。
